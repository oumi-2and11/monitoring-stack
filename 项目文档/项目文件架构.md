## 2. 作业要求逐条“对标”怎么满足（稳妥版）

### (1) “搭建 Prometheus+Grafana 全方位监控告警系统”

你用 docker compose 起：

- Prometheus（采集 + rules）
- Grafana（展示）
- Alertmanager（告警通知）
- node_exporter（主机指标）
- cAdvisor（Docker 容器指标）
- Flask app（你自己写的服务指标）
- alert_receiver（你自己写的 webhook 接收器，用于演示告警落地）

> 这条是工程落地的主干。

------

### (2) “静态 + 动态服务发现”

- **静态发现**：prometheus.yml 里写死 targets（例如 node_exporter、cadvisor、flask_app）
- **动态发现（稳妥版强推 file_sd_configs）**：
	- Prometheus 支持从一个 JSON 文件读取 targets 列表
	- 你把 `prometheus/file_sd/targets.json` 提交进去
	- 你写一个小脚本（可选）来“改 targets.json 并让 Prometheus 自动刷新”，这就叫动态发现（不用硬上 k8s / docker sd）

> 评分点：你在报告里解释为什么容器环境必须动态发现、file_sd 的工作机制和优缺点。

------

### (3) “对容器、物理节点、service、pod 等资源指标监控”

稳妥版的落地做法：

- **物理节点（本机）**：node_exporter（CPU、内存、磁盘、网络）
- **容器**：cAdvisor（每个容器 CPU/内存/网络/重启等）
- **service（你的服务）**：Flask + prometheus_client（QPS、错误率、延迟）
- **pod**：这一项你可以用**研究为辅**完成：
	- 报告里写 K8s 下如何用 kube-state-metrics、cAdvisor、kubelet、metrics-server 采集 pod 指标
	- 给出 Prometheus 在 K8s 的 service discovery（基于 Kubernetes API）怎么配置
	- 你可以附“如果有环境，可用 minikube 复现”的步骤（可选）

> 老师一般会认可这种“Docker落地 + K8s路径清晰可复现”的完成方式，尤其你是单人。

------

### (4) “Grafana web界面展示 Prometheus 监控指标”

- 你必须做至少 2 个 dashboard：
	1. Host & Docker 概览（node_exporter + cadvisor）
	2. Flask 服务概览（自定义业务指标）
- **高分关键**：把 dashboard 导出成 JSON 并通过 provisioning 自动导入（老师不用手动配置）

------

### (5) “研究 docker 与 kubernetes 编排环境下的安全监控”

这部分主要体现在报告里，你可以写得很扎实，建议包含：

- **监控系统本身的安全风险**：
	- /metrics 暴露信息（进程、版本、路径等），可能造成信息泄露
	- Prometheus/Grafana/Alertmanager 默认无强认证，暴露端口的风险
	- 解决：网络隔离（只在内网/compose网络）、反向代理 + BasicAuth、TLS、Grafana 权限、最小暴露端口
- **容器/编排环境的安全监控思路**：
	- 异常重启次数、异常网络流量、错误率突增、CPU/内存异常飙升作为安全事件“间接指标”
	- 用告警规则做初步检测（不是替代 IDS，但能做 SRE+安全联动）

------

### (6) “研究 Prometheus 告警工具包 + K8s下部署”

- 告警工具包就是 Prometheus rules + Alertmanager 路由/接收器
- K8s 部署写清楚两条路线：
	1. Helm 安装 kube-prometheus-stack（最常见）
	2. Prometheus Operator 的 CRD 方式（ServiceMonitor/PodMonitor）
- 你可以画一张“Docker版 vs K8s版”对比表（采集方式、服务发现、运维复杂度、安全边界）

## 3. Windows 环境的特别注意（避免踩坑）

1. **建议使用 Docker Desktop（WSL2 后端）**，这样 cAdvisor 挂载路径更稳定
2. cAdvisor 在 Windows 上有时对宿主机指标不如 Linux 完整，但监控“容器层面”通常够用
3. node_exporter 是给 Linux 用的；在 Windows 上严格来说应该用 `windows_exporter`。
	- 但很多课程验收并不会纠结你宿主机是 Windows 还是 Linux
	- **稳妥策略**：你可以把“node_exporter”理解为“节点 exporter”，并在报告里说明 Windows 环境可替换为 windows_exporter（加分点：体现严谨）
4. 所有配置里 targets 用容器服务名（如 `prometheus:9090`、`cadvisor:8080`），不要写 `localhost`

> 如果你愿意，我也可以按“Windows + windows_exporter”给你一个更严谨的版本（更贴合“物理节点”监控）。
