# 项目文件架构

## 0. 最终你交付给老师的"成果长什么样"

老师解压你的压缩包后：

1. 运行：

```powershell
docker compose up -d --build
```

2. 访问：

- Grafana：`http://localhost:3000`（admin/admin 登录，直接看到做好的 Dashboards）
- Prometheus：`http://localhost:9090`（Targets 全是 UP，Rules 已加载）
- Alertmanager：`http://localhost:9093`（能看到告警分组/告警）
- Flask 被测服务：`http://localhost:8000`（以及 `http://localhost:8000/metrics`）
- alert_receiver：`http://localhost:5001`（查看已接收的告警 JSON）
- cAdvisor：`http://localhost:8080`（容器指标 UI）

3. 运行脚本触发告警（`scripts/trigger_error.ps1`），录屏展示：

- Grafana 面板里错误率上升
- Prometheus "ALERTS" 变成 FIRING
- Alertmanager 出现告警
- alert_receiver 页面收到告警 JSON

这就是高分"工程闭环"。

## 2. 作业要求逐条"对标"怎么满足

### (1) "搭建 Prometheus+Grafana 全方位监控告警系统"

docker compose 起 7 个服务：

- Prometheus（采集 + rules）
- Grafana（展示）
- Alertmanager（告警通知）
- node_exporter（主机指标）
- cAdvisor（Docker 容器指标）
- Flask app（自研服务指标）
- alert_receiver（自研 webhook 接收器，告警落地展示）

> 这条是工程落地的主干。

------

### (2) "静态 + 动态服务发现"

- **静态发现**：prometheus.yml 里写死 targets（node_exporter、cadvisor、flask_app）
- **动态发现（file_sd_configs）**：
	- Prometheus 从 `file_sd/targets.json` 读取 targets 列表
	- 修改 JSON 文件后 Prometheus 30s 内自动刷新（`refresh_interval: 30s`）
	- `scripts/add_target.ps1` 可动态追加目标

> 评分点：报告里解释为什么容器环境必须动态发现、file_sd 的工作机制和优缺点。

------

### (3) "对容器、物理节点、service、pod 等资源指标监控"

落地做法：

- **物理节点**：node_exporter（CPU、内存、磁盘、网络）
	- Docker Desktop 运行在 WSL2 Linux 内核上，容器内用 node_exporter 采集
	- 报告里补充说明：Windows 原生环境应替换为 windows_exporter，两者指标命名有差异（如 `windows_cpu_time_total` vs `node_cpu_seconds_total`），PromQL 需适配
- **容器**：cAdvisor（每个容器 CPU/内存/网络/重启等）
	- 注意：cAdvisor 在 Windows Docker Desktop (WSL2) 下可能无法获取完整 cgroup 指标，报告里说明此限制
- **service（你的服务）**：Flask + prometheus_client（QPS、错误率、延迟）
- **pod**：这一项用**研究为辅**完成：
	- 报告里写 K8s 下如何用 kube-state-metrics、cAdvisor、kubelet、metrics-server 采集 pod 指标
	- 给出 Prometheus 在 K8s 的 service discovery（基于 Kubernetes API）怎么配置
	- 附"如果有环境，可用 minikube 复现"的步骤（可选）

> 老师一般会认可这种"Docker落地 + K8s路径清晰可复现"的完成方式，尤其你是单人。

### (5) "研究 docker 与 kubernetes 编排环境下的安全监控"

这部分**报告 + 实操**都要做：

#### 报告部分

- **监控系统本身的安全风险**：
	- /metrics 暴露信息（进程、版本、路径等），可能造成信息泄露
	- Prometheus/Grafana/Alertmanager 默认无强认证，暴露端口的风险
	- 解决：Docker 网络隔离、Grafana 权限控制、最小暴露端口
- **容器/编排环境的安全监控思路**：
	- 异常重启次数、异常网络流量、错误率突增、CPU/内存异常飙升作为安全事件"间接指标"
	- 用告警规则做初步检测（不是替代 IDS，但能做 SRE+安全联动）

#### 实操部分（安全课高分关键）

- **Docker 网络隔离**：
	- docker-compose.yml 里定义 `monitoring-net` 桥接网络，所有监控组件只在此内部网络通信
	- 报告里画网络拓扑图，标注哪些端口对外、哪些仅内部
- **Grafana 权限控制**：
	- 配置 Admin + Viewer 两个角色，演示最小权限原则
- **告警规则作为安全检测**：
	- ContainerRestartCount（异常重启）、FlaskAppErrorRateHigh（错误率突增）可作为安全事件"间接指标"

> 这是网络空间安全课的作业，安全实操的权重比普通课程高。

### (6) "研究 Prometheus 告警工具包 + K8s下部署"

- 告警工具包就是 Prometheus rules + Alertmanager 路由/接收器
- K8s 部署写清楚两条路线：
	1. Helm 安装 kube-prometheus-stack（最常见）
	2. Prometheus Operator 的 CRD 方式（ServiceMonitor/PodMonitor）
- 可以画一张"Docker版 vs K8s版"对比表（采集方式、服务发现、运维复杂度、安全边界）

#### 告警规则（5 条）

| 规则名 | 表达式 | 阈值 | 持续时间 | 级别 |
|---|---|---|---|---|
| HostHighCpuLoad | `100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[2m])) * 100)` | > 80% | 1m | warning |
| HostHighMemoryUsage | `100 - (100 * node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)` | > 90% | 2m | warning |
| FlaskAppErrorRateHigh | `sum by(job,instance)(rate(flask_http_request_total{status_code=~"5.."}[1m])) / sum by(job,instance)(rate(flask_http_request_total[1m])) * 100` | > 5% | 1m | critical |
| ContainerHighCpu | `container_cpu_usage_seconds_total` 按容器聚合 | > 80% | 2m | warning |
| ContainerRestartCount | `increase(container_restart_count[10m])` | > 3 | 0m | critical |

## 3. Windows 环境的特别注意

1. **使用 Docker Desktop（WSL2 后端）**
2. cAdvisor 在 Windows 上对宿主机指标不如 Linux 完整，但监控"容器层面"通常够用
3. **node_exporter vs windows_exporter**：
	- Docker Desktop (WSL2) 实际运行在 Linux 内核上，容器内使用 node_exporter
	- 如果要在 Windows 宿主机上直接采集（不经过 WSL2），应安装 windows_exporter（MSI 包）
	- 报告里说明两者差异，体现"因地制宜"（加分点）
4. 所有配置里 targets 用容器服务名（如 `prometheus:9090`、`cadvisor:8080`），不要写 `localhost`
5. **数据持久化**：docker-compose.yml 配置了 volumes，防止容器重启后数据丢失：
	- `prometheus_data:/prometheus`
	- `grafana_data:/var/lib/grafana`
	- `alertmanager_data:/alertmanager`
