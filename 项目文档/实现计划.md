# 实现计划

## 阶段概览

| 阶段 | 内容 | 预计耗时 |
|---|---|---|
| 一 | 基础骨架：Docker Compose + Prometheus + Grafana 跑通 | 1天 |
| 二 | 采集层：windows_exporter + cAdvisor + Flask 服务指标 | 1天 |
| 三 | 服务发现：静态配置 + file_sd 动态发现 | 半天 |
| 四 | 告警体系：Prometheus Rules + Alertmanager + webhook receiver | 1天 |
| 五 | 可视化：Grafana Dashboard + Provisioning | 1天 |
| 六 | 安全加固：Nginx 反向代理 + BasicAuth + 网络隔离 | 1天 |
| 七 | 压测与验收：Locust 压测 + 全链路验证 + 脚本 | 1天 |

## 阶段一：基础骨架

### 开发内容

1. 创建 `docker-compose.yml`，定义以下服务：
   - prometheus（端口 9090）
   - grafana（端口 3000）
2. 创建 `prometheus/prometheus.yml`，最简配置：
   - global scrape_interval 15s
   - 一个 static_configs job，target 指向 prometheus 自身（`localhost:9090`）
3. 创建 `grafana/provisioning/datasources/datasource.yml`：
   - 类型 prometheus，URL `http://prometheus:9090`
4. 配置 volumes 持久化：
   - `prometheus_data:/prometheus`
   - `grafana_data:/var/lib/grafana`

### 验证方法

```powershell
docker compose up -d
# 1. 访问 http://localhost:9090/targets，确认 prometheus 自身 target 状态为 UP
# 2. 访问 http://localhost:3000，确认 Grafana 能登录（默认 admin/admin）
# 3. Grafana → Configuration → Data Sources，确认 Prometheus 数据源已自动配置且连通
# 4. 在 Grafana Explore 里执行 up{job="prometheus"}，确认返回结果
docker compose down
```

## 阶段二：采集层

### 开发内容

1. **windows_exporter**
   - 在 `docker-compose.yml` 中添加 windows_exporter 服务（端口 9182）
   - 在 `prometheus/prometheus.yml` 中添加 static_configs job，target 指向 `windows_exporter:9182`
2. **cAdvisor**
   - 在 `docker-compose.yml` 中添加 cAdvisor 服务（端口 8080）
   - 挂载 `/var/lib/docker`、`/sys/fs/cgroup`、`/var/run/docker.sock`（只读）
   - 在 `prometheus/prometheus.yml` 中添加 static_configs job，target 指向 `cadvisor:8080`
3. **Flask 应用**
   - 创建 `services/flask_app/app.py`：
     - 使用 `prometheus_client` 暴露 Counter（请求总数、错误数）和 Histogram（请求延迟）
     - 提供 `/`（正常接口）、`/error`（模拟 500 错误）、`/slow`（模拟慢请求）、`/metrics`（指标端点）
   - 创建 `services/flask_app/requirements.txt`、`Dockerfile`
   - 在 `docker-compose.yml` 中添加 flask_app 服务（端口 8000）
   - 在 `prometheus/prometheus.yml` 中添加 static_configs job，target 指向 `flask_app:8000`

### 验证方法

```powershell
docker compose up -d --build
# 1. 访问 http://localhost:9090/targets，确认三个新 target 全部 UP
# 2. 访问 http://localhost:8000/metrics，确认能看到 flask_http_request_* 指标
# 3. 访问 http://localhost:8000/ 和 /error，再刷新 /metrics，确认 Counter 在增长
# 4. 在 Prometheus UI 查询 windows_cpu_time_total，确认有数据
# 5. 在 Prometheus UI 查询 container_cpu_usage_seconds_total，确认 cAdvisor 有数据
docker compose down
```

## 阶段三：服务发现

### 开发内容

1. **静态发现**（阶段二已完成）
   - prometheus.yml 中的 static_configs 就是静态发现
2. **动态发现（file_sd_configs）**
   - 创建 `prometheus/file_sd/targets.json`，内容为一个 job 的 targets 列表（例如把 flask_app 的 target 放进这个文件）
   - 在 `prometheus/prometheus.yml` 中添加一个新 job，使用 `file_sd_configs` 指向 `file_sd/targets.json`
   - 创建 `scripts/add_target.ps1`：往 targets.json 追加一个新 target，演示 Prometheus 自动感知

### 验证方法

```powershell
docker compose up -d
# 1. 访问 http://localhost:9090/targets，确认 file_sd 类型的 job 出现且 target 为 UP
# 2. 运行 scripts/add_target.ps1 往 targets.json 加一个新 target
# 3. 等待约 30 秒（Prometheus 默认刷新间隔），刷新 /targets 页面
# 4. 确认新 target 自动出现（状态可能是 DOWN，因为服务不存在，但说明发现机制生效）
# 5. 在 Prometheus UI 查询 up{job="flask_app_sd"}，确认 file_sd 采集正常
docker compose down
```

## 阶段四：告警体系

### 开发内容

1. **Prometheus 告警规则**
   - 创建 `prometheus/rules/alerts.yml`，包含 5 条规则：
     - HostHighCpuLoad（CPU > 80% 持续 1m）
     - HostHighMemoryUsage（内存 > 90% 持续 2m）
     - FlaskAppErrorRateHigh（5xx 错误率 > 5% 持续 1m）
     - ContainerHighCpu（容器 CPU > 80% 持续 2m）
     - ContainerRestartCount（10 分钟内重启 > 3 次）
   - 在 `prometheus/prometheus.yml` 中添加 `rule_files` 指向 `rules/alerts.yml`
2. **Alertmanager**
   - 创建 `alertmanager/alertmanager.yml`：
     - route：按 job 分组，默认 receiver 为 webhook
     - receivers：webhook 指向 `http://alert_receiver:5001/webhook`
   - 在 `docker-compose.yml` 中添加 alertmanager 服务（端口 9093）
   - 在 `prometheus/prometheus.yml` 中添加 `alerting.alertmanagers` 配置
   - 配置 `alertmanager_data:/alertmanager` 持久化
3. **alert_receiver（webhook 接收器）**
   - 创建 `services/alert_receiver/app.py`：
     - Flask 服务，接收 POST /webhook，将告警 JSON 存入内存列表
     - GET / 页面展示已收到的所有告警
   - 创建 `services/alert_receiver/requirements.txt`、`Dockerfile`
   - 在 `docker-compose.yml` 中添加 alert_receiver 服务（端口 5001）
4. **触发脚本**
   - 创建 `scripts/trigger_error.ps1`：循环调用 `/error` 接口，拉高错误率触发告警
   - 创建 `scripts/trigger_latency.ps1`：循环调用 `/slow` 接口，模拟延迟

### 验证方法

```powershell
docker compose up -d --build
# 1. 访问 http://localhost:9090/rules，确认 5 条规则全部加载
# 2. 访问 http://localhost:9090/alerts，确认所有规则状态为 INACTIVE（绿色）
# 3. 访问 http://localhost:9093，确认 Alertmanager UI 正常
# 4. 运行 scripts/trigger_error.ps1（持续约 2 分钟）
# 5. 刷新 http://localhost:9090/alerts，确认 FlaskAppErrorRateHigh 状态变为 FIRING（红色）
# 6. 访问 http://localhost:9093，确认 Alertmanager 中出现该告警
# 7. 访问 http://localhost:5001，确认 alert_receiver 页面显示了告警 JSON 内容
# 8. 停止触发脚本，等待告警恢复（RESOLVED），确认 alert_receiver 收到恢复通知
docker compose down
```

## 阶段五：可视化

### 开发内容

1. **Dashboard 1：Host & Docker 概览**
   - 在 Grafana 手动创建 Dashboard，包含以下 Panel：
     - CPU 使用率（windows_exporter）
     - 内存使用率（windows_exporter）
     - 磁盘使用率（windows_exporter）
     - 网络流量（windows_exporter）
     - 容器 CPU Top5（cAdvisor）
     - 容器内存 Top5（cAdvisor）
     - 容器重启次数（cAdvisor）
   - 导出 JSON 保存为 `grafana/dashboards/01-host-and-docker.json`
2. **Dashboard 2：Flask 服务概览**
   - 在 Grafana 手动创建 Dashboard，包含以下 Panel：
     - 请求 QPS（按状态码）
     - 错误率（5xx 占比）
     - 请求延迟 P50/P95/P99
     - 告警状态（ALERTS 指标）
   - 导出 JSON 保存为 `grafana/dashboards/02-flask-service.json`
3. **Provisioning 自动导入**
   - 创建 `grafana/provisioning/dashboards/providers.yml`：
     - 指向 `/var/lib/grafana/dashboards` 目录
     - 自动加载该目录下所有 JSON 文件
   - 在 `docker-compose.yml` 中将 `grafana/dashboards/` 挂载到容器内对应路径

### 验证方法

```powershell
docker compose down -v   # 清除旧数据，模拟全新环境
docker compose up -d
# 1. 访问 http://localhost:3000，确认无需手动配置
# 2. 左侧菜单 → Dashboards，确认 2 个 Dashboard 自动出现
# 3. 打开 "Host & Docker 概览"，确认所有 Panel 有数据（非 No Data）
# 4. 打开 "Flask 服务概览"，先用 curl 访问几次 / 和 /error，确认 Panel 有数据
# 5. 修改 Dashboard JSON 中某个 Panel 的标题，重启 Grafana，确认修改生效（验证 Provisioning 机制）
docker compose down
```

## 阶段六：安全加固

### 开发内容

1. **Docker 网络隔离**
   - 在 `docker-compose.yml` 中定义内部网络 `monitoring-net`（internal: true）
   - 所有监控组件（windows_exporter、cAdvisor、flask_app、alert_receiver）只加入 `monitoring-net`
   - 定义外部网络 `frontend-net`
   - Nginx 同时加入 `frontend-net` 和 `monitoring-net`
   - **移除所有服务直接暴露的宿主机端口**，只保留 Nginx 的 80 端口映射
2. **Nginx 反向代理**
   - 创建 `nginx/nginx.conf`：
     - `/prometheus/` → `http://prometheus:9090/`
     - `/grafana/` → `http://grafana:3000/`（需配置 Grafana 的 root_url）
     - `/alertmanager/` → `http://alertmanager:9093/`
     - 所有 location 启用 `auth_basic`，密码文件为 `nginx/.htpasswd`
   - 创建 `nginx/.htpasswd`（用 htpasswd 工具生成）
   - 在 `docker-compose.yml` 中添加 nginx 服务，挂载配置文件
3. **Grafana 权限控制**
   - 配置 admin 密码（环境变量 GF_SECURITY_ADMIN_PASSWORD）
   - 通过 Grafana API 或 provisioning 创建一个 Viewer 角色，只读权限
4. **Prometheus /metrics 加密验证**
   - 通过 Nginx 代理后，直接访问 `http://localhost:9090/metrics` 应不可达
   - 只能通过 `http://localhost/prometheus/metrics` + BasicAuth 访问

### 验证方法

```powershell
docker compose up -d --build
# 1. 直接访问 http://localhost:9090，确认连接被拒绝（端口不再暴露）
# 2. 直接访问 http://localhost:3000，确认连接被拒绝
# 3. 无密码访问 http://localhost/prometheus/，确认返回 401
# 4. 带密码访问 http://localhost/prometheus/，确认正常看到 Prometheus UI
# 5. 带密码访问 http://localhost/grafana/，确认正常看到 Grafana UI
# 6. 带密码访问 http://localhost/alertmanager/，确认正常看到 Alertmanager UI
# 7. 用 Viewer 账号登录 Grafana，确认无法修改 Dashboard 设置
# 8. 在报告里截图对比：加固前（所有端口可直连）vs 加固后（统一入口 + 认证）
docker compose down
```

## 阶段七：压测与验收

### 开发内容

1. **Locust 压测**
   - 创建 `loadtest/locustfile.py`：
     - 定义用户行为：80% 访问 `/`，10% 访问 `/error`，10% 访问 `/slow`
   - 在 `docker-compose.yml` 中添加 locust 服务（或本地运行）
2. **启动/停止脚本**
   - 创建 `scripts/up.ps1`：`docker compose up -d --build`，等待所有服务健康检查通过
   - 创建 `scripts/down.ps1`：`docker compose down`，可选 `-v` 清数据
3. **全链路验证脚本**（可选）
   - 创建一个验证脚本，自动检查所有服务状态、target 状态、dashboard 是否加载

### 验证方法

```powershell
# 压测验证
docker compose up -d --build
# 1. 启动 Locust：locust -f loadtest/locustfile.py --host=http://localhost:8000
# 2. 打开 Locust UI (http://localhost:8089)，设置 50 用户，启动压测
# 3. 压测 3-5 分钟，观察：
#    - Grafana "Flask 服务概览" Dashboard 中 QPS、延迟、错误率实时变化
#    - Prometheus /alerts 中告警状态变化
#    - Alertmanager 和 alert_receiver 收到告警
# 4. 停止压测，截图 Grafana 面板（写入报告"实验与评估"章节）

# 全链路验收
.\scripts\up.ps1
# 1. 确认所有容器 running：docker compose ps
# 2. 确认所有 Prometheus targets UP：curl http://localhost/prometheus/api/v1/targets -u user:pass
# 3. 确认 2 个 Dashboard 已加载
# 4. 运行 trigger_error.ps1，确认告警全链路触发
# 5. 运行 trigger_latency.ps1，确认延迟告警触发
# 6. 确认安全加固生效（无密码访问被拒）
.\scripts\down.ps1
```

```
阶段一 ──► 阶段二 ──► 阶段三
              │
              ├──► 阶段四
              │
              ├──► 阶段五
              │
              └──► 阶段六

阶段四 + 阶段五 + 阶段六 ──► 阶段七
```

- 阶段一是基础，必须先完成
- 阶段二完成后，阶段三、四、五、六可以并行开发（但建议按顺序，避免配置冲突）
- 阶段七必须在所有功能完成后进行
